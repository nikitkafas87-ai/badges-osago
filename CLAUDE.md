# Claude Code — Правила исполнительной дисциплины

> Положи этот файл в корень проекта или в `~/.claude/CLAUDE.md` для глобального действия.
> Эти правила не дают Claude молча ломаться, уходить от плана и тратить время на сломанные конвейеры.

## 1. Проверяй каждый шаг

После каждого значимого действия — ПРОВЕРЬ результат перед переходом к следующему:
- Создал файл? Прочитай его и убедись в содержимом.
- Записал данные в БД/таблицу/API? Запроси обратно и сверь.
- Запустил скрипт? Проверь И exit code, И содержимое результата.
- Задеплоил что-то? Убедись что оно реально доступно.

**Никогда не считай шаг выполненным без проверки.** "Нет ошибки в выводе" ≠ "всё сработало".

## 2. Целостность конвейера

При выполнении многоэтапных задач (A → B → C → D):
- Если шаг C сломался — СТОП. Не продолжай D. Не перезапускай A и B.
- Перед каждым новым шагом проверь что ПРЕДЫДУЩИЙ реально завершён успешно.
- Каждые 10-15 итераций батч-операции — проверяй что весь конвейер работает end-to-end.
- Если обнаружил что промежуточный шаг молча ломается — ОСТАНОВИ всю работу и сообщи пользователю.

## 3. Верность плану

Когда план согласован с пользователем:
- **Следуй плану шаг за шагом.** Не перескакивай, не меняй порядок без причины.
- **Если шаг не получается — ретрай тем же способом** (2-3 попытки), а не переключайся на другой подход.
- **Если после ретраев не работает — СПРОСИ пользователя.** Не изобретай альтернативный путь самостоятельно.
- **НИКОГДА не возвращайся к подходу, который уже был отвергнут** в текущей сессии. Если путь X не сработал — не пробуй X снова.
- Если вынужден отклониться от плана — СНАЧАЛА сообщи пользователю что и почему, ПОТОМ действуй.

## 4. Контрольные точки в длинных задачах

При задачах с >10 действиями:
- Веди счётчик: "Шаг 5/12: загрузка изображений в хранилище"
- Каждые 5 шагов — краткий статус пользователю: что сделано, что впереди, есть ли проблемы.
- При батч-операциях (100+ элементов) — промежуточный отчёт каждые 20-30 элементов.

## 5. Обнаружение молчаливых сбоев

Красные флаги — ОСТАНОВИСЬ и проверь:
- Один и тот же вызов возвращает один и тот же результат 3+ раза подряд
- Размер результата резко изменился (было 50 элементов, стало 0)
- Операция "успешна", но при проверке данных нет
- Таймаут или пустой ответ от внешнего сервиса
- Повторяющиеся ошибки 4xx/5xx

При обнаружении: СТОП → диагностика → сообщить пользователю → ждать решения.

## 6. Без тупых циклов

- Не повторяй одну и ту же падающую операцию больше 3 раз.
- Не делай sleep-и-retry в цикле в надежде что проблема рассосётся.
- Если после 2-3 попыток не работает — проблема структурная. Разбирайся или спрашивай.

## 7. Дисциплина объёма

- Делай только то, что просили. Не "улучшай" окружающий код, не добавляй фичи, не рефактори без запроса.
- Не добавляй обработку ошибок для сценариев, которые не могут произойти.
- Не создавай абстракции для одноразовых операций.
- Три похожих строки кода лучше преждевременной абстракции.

## 8. Расположение скилов и настроек

Все скилы, настройки и конфигурация хранятся **только в проекте** (`autotests/.claude/`), не в глобальной системе (`~/.claude/`).

- Скилы → `autotests/.claude/skills/[имя скила]/SKILL.md`
- Разрешения MCP и прочее → `autotests/.claude/settings.local.json`

Если не уверен куда сохранять — **спроси пользователя**, не пиши в глобальный `~/.claude/`.

## 9. Авто-именование сессии

При старте КАЖДОЙ новой сессии — сразу после получения первого сообщения пользователя, перед любым другим действием:
1. Проанализируй первый запрос и сформулируй короткое название сессии (2-4 слова на русском), отражающее суть задачи.
2. Выполни bash-команду: printf '\033]0;Название сессии\007' — это задаст заголовок вкладки терминала.
3. Делай молча, без анонса. Название должно быть полезным для навигации между вкладками.

Примеры названий:
- «Отчёт МФО Локо» — если просят отчёт по партнёру
- «Настройка Claude Code» — если настраивают окружение
- «Баг авторизации» — если чинят баг
- «Сверка февраль» — если делают сверку
- «GitLab отчёт команды» — если просят отчёт по разработчикам

## 10. Расшифровка enum-ов из БД

При встрече любого числового типа в БД (например `OfferStatusTypeId=6`, `UpsaleType=5`, `ApplicationTypeId=3` и т.д.) — **всегда читать соответствующий файл** из папки:
```
C:\Users\nikit\projects\autotests\Enum_extracted\
```

- Имя файла = имя enum-а. Например: `OfferStatusType.cs`, `UpsaleType.cs`, `ApplicationType.cs`.
- Делать проактивно, без напоминания пользователя.
- Если файл не найден — использовать глоссарий БД или спросить пользователя.
