# Отчёт по сотруднику в Яндекс Трекере

**Команда:** `/tracker_report_active [имя] [период]`

Примеры:
- `/tracker_report_active Котов`
- `/tracker_report_active Свистунов 2–10 марта`
- `/tracker_report_active Листопад март 2026`

---

## Алгоритм

### Шаг 1. Разбери аргументы

- Имя сотрудника (обязательно)
- Период (опционально, по умолчанию — текущая неделя)
- Если период не задан явно, используй последние 7 дней от сегодня

### Шаг 2. Запусти параллельно

1. `mcp__tracker__get_worklogs` — залогированные часы за период
2. `mcp__tracker__search_issues` — задачи с assignee = новый аккаунт (newId), фильтр `updatedAt`
3. `mcp__tracker__search_issues` — задачи с assignee = старый логин, фильтр `updatedAt`

Для поиска используй `mcp__tracker__resolve_user` если нужно получить оба аккаунта.

Фильтр для поиска задач:
```json
{"assignee": "<id_или_логин>", "updatedAt": {"from": "<from>T00:00:00", "to": "<to>T23:59:59"}}
```

### Шаг 3. Сформируй отчёт

**Блок 1 — Залогированное время** (показывай только если totalHours > 0):
```
## ⏱ Залогировано: X часов

| Задача | Часов | Комментарий |
|--------|-------|-------------|
| INS-123 | 2.5 | описание работы |
```

Если totalHours = 0 — напиши: `Время не залогировано за период.`

**Блок 2 — Активные задачи** (дедуплицируй по ключу из обоих поисков):

Группируй по статусу:

```
## 📋 Задачи: X шт.

**В работе (N):**
- INS-XXX — название задачи [дата обновления]

**На тестировании (N):**
- INS-XXX — название задачи [дата обновления]

**Закрыто (N):**
- INS-XXX — название задачи [дата обновления]

**Прочие (N):**
- INS-XXX — название задачи [дата обновления]
```

**Блок 3 — Итог:**
```
---
Сотрудник: [Имя]  |  Период: [период]  |  Задач: X  |  Закрыто: Y  |  Часов: Z
```

---

## Правила

- Всегда ищи по обоим аккаунтам (resolve_user даёт old + new)
- Дедуплицируй задачи по ключу (INS-XXX)
- Если задач нет совсем — сообщи об этом явно
- Не показывай backlog-задачи без обновлений за период, если их слишком много (>10) — сверни в "Backlog: N задач"
- Используй эмодзи умеренно, только в заголовках блоков

---

## Критические ловушки

### ❌ НЕ используй get_employee_stats / get_team_stats для анализа периода
Эти инструменты фильтруют по **createdAt** (дата создания задачи), а не по дате работы.
Они покажут задачи созданные в период, но не задачи над которыми работали в период.
**Всегда используй** `search_issues` с фильтром `updatedAt`.

### ⚠️ Миграция логинов — всегда проверяй оба
`resolve_user` возвращает `oldLogin` и `newId`. Ищи по обоим параллельно.
Иногда "новый" логин существует в системе, но задач на него нет (сотрудник работал под старым).
Иногда "новый" логин вовсе не существует — тогда активным является "старый".

**Пример:** `resolve_user` вернул `new_login` как активный, но он возвращает 0 задач — возможно, реально активным остался `old_login`.
Если один логин возвращает 0 задач — это не значит что человек ничего не делал. Проверь второй.

### Stale-задачи
Stale = задачи в активных статусах (В работе, На тестировании), не обновлявшиеся >7 дней.
Добавляй счётчик stale к итоговой строке если >0:
```
Сотрудник: [Имя]  |  Период: [период]  |  Задач: X  |  Закрыто: Y  |  Stale: Z  |  Часов: W
```
